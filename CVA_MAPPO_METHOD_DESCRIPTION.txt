CVA-MAPPO 方法说明文档
======================

一、研究背景
------------

多星对地观测任务调度需要在有限卫星资源、有限可见时间窗口、姿态机动约束和任务截止时间约束下,为大量观测任务安排执行卫星和执行时刻。真实场景中任务通常同时包含常规任务和动态突发任务,例如灾害、冲突、热点事件等。动态任务具有到达时间不确定、响应时效要求高、可见窗口有限等特点,这使得调度问题不再是一个静态离线优化问题,而是一个大规模、多智能体、动态决策问题。

原始 MRL-DMS 思路使用 PPO/LSTM 等强化学习方法进行任务调度,能够体现单星或多星策略学习能力。但在任务规模扩大后,直接让 PPO/MAPPO 面对全部任务动作空间会出现明显瓶颈。例如在 12 星、1200 个常规任务、3 次动态插入且每次 300 个动态任务的压力场景中,总任务槽位达到 2100。若低层策略直接输出 2100+1 维离散动作,不仅计算代价高,而且探索空间过大,策略很难稳定学习到高价值任务选择规律。

因此,本文工作的核心背景是:在大规模多星动态任务调度中,单纯扩大 PPO/MAPPO 的动作空间并不能自然获得可扩展的协同调度能力,必须引入高层任务分配与低层强化学习调度的分层机制。


二、要解决的问题
----------------

本文方法主要解决以下问题:

1. 大动作空间导致 PPO/MAPPO 学习困难

当任务数量达到数千级时,每颗卫星直接在全部任务槽位中选择动作会造成动作维度过大、有效动作稀疏、探索效率低、训练时间长等问题。

2. 多星独立决策容易产生重复观测

如果多颗卫星独立选择任务,它们可能选择同一个高价值任务,造成重复观测和资源浪费。即使总完成任务数较高,也可能包含大量重复调度,不能体现真正的协同规划能力。

3. 静态任务分配难以适应动态任务和 owner 失效

一次性为任务分配负责卫星可以降低重复观测,但动态任务到达、任务临近截止、原 owner 不再具备未来可行窗口等情况会导致静态分配失效。如果不重新调整 owner,会损失任务完成率和动态响应能力。

4. 只用启发式分配缺少上下文价值判断

简单按照可见质量、负载或截止时间进行任务分配,难以同时考虑任务集合结构、任务稀缺性、动态任务优先级、未来窗口和多星负载状态。需要一个能够综合上下文信息的价值感知分配模块。


三、本文方法: CVA-MAPPO
----------------------

本文最终方案定义为 CVA-MAPPO:

    CVA-MAPPO =
    Context-Value-Aware Assignment
    + Top-K Candidate Action Space
    + Rolling Owner Reallocation
    + Low-level MAPPO Scheduling

整体思想是:高层模块先从全量任务中进行价值感知任务分配和候选筛选,低层 MAPPO 只在每颗卫星的小规模候选任务集合中进行时序调度决策。

1. 高层上下文价值感知任务分配

高层模块为每个任务分配一个默认负责卫星 owner。分配分数综合考虑:

- 卫星对任务的可见窗口质量,如 off-nadir 角;
- 任务优先级;
- 动态任务标记;
- deadline 压力;
- 任务可由多少颗卫星完成,即任务稀缺性;
- 当前卫星负载和目标负载比例;
- owner 是否已经失效;
- owner 切换历史;
- deadline release 和 rescue 情况;
- LSTM/GRU/Transformer/Set Transformer 等上下文编码器给出的任务集合价值信息。

该模块不是简单的最近窗口贪心分配,而是一个上下文价值感知的任务-owner 分配器。它的作用是把全局任务集合初步拆分给不同卫星,从源头减少多星重复观测。

2. Top-K 候选动作空间压缩

在压力场景中,底层 MAPPO 不再直接面对 2100 个任务动作。环境仍保留全部真实任务槽位用于统计、可见窗口计算、owner 分配和结果可视化,但对每颗卫星的策略网络只暴露:

    Top-K 候选任务 + idle

当前推荐设置为:

    candidate_action_top_k = 128

Top-K 候选集的构造方式是:

- 优先放入当前可执行任务;
- 若当前可执行任务不足 K 个,用未来高价值 owner 任务补齐观测槽;
- mask 仍然只允许当前可执行任务被执行;
- 未来任务进入候选观测,主要帮助策略判断是否应该 idle 等待更高价值窗口。

这样做可以同时保留全局任务统计口径和低层策略学习效率。PPO/MAPPO 的 actor 输出维度从约 2101 降低到 129,明显降低训练和探索难度。

3. 滚动 owner 重分配与 rescue 机制

静态 owner 分配并不足以适应动态任务场景。因此方法加入 rolling reallocation:

- 动态任务到达时可以触发增量分配;
- 周期性检查 owner 是否仍具备未来可行窗口;
- 任务临近 deadline 时允许非 owner 接手;
- owner 失效时允许其他卫星 rescue;
- owner 切换受到 switch penalty 和 lock window 约束,避免频繁无意义切换。

因此,owner 不是绝对不可变的硬约束,而是默认责任分配。deadline release 和 stale owner rescue 是容错机制,用于提升动态任务完成率和末段任务利用率。

4. 低层 MAPPO 协同调度

低层 MAPPO 在每颗卫星的 Top-K 候选空间内进行动作选择。MAPPO 保留集中式训练、分布式执行思想:

- actor 使用每颗卫星的局部候选任务观测;
- critic 使用全局状态或全局统计信息;
- 多星执行后进行冲突解决;
- 一颗卫星完成任务后,任务状态同步到所有卫星,避免重复观测。

如果多颗卫星由于 release/rescue 同时选择同一任务,协同冲突解决模块只允许价值最高或最合适的一颗卫星执行,其他卫星 idle 或在评估阶段改派到次优任务。因此即使存在临时接手机制,最终仍能控制重复观测率。


四、如何证明方法有效
--------------------

本文需要通过主对比、消融实验和扩展性实验证明方法确实解决了上述问题。

所有正式压力实验建议使用预生成场景缓存:

- train_scenarios.pkl:训练场景池,按简单到复杂 curriculum 组织;
- eval_scenarios.pkl:独立评估场景池,使用不同 seed,不与训练场景混用;
- vtw_cache:预热后的卫星-目标可见时间窗口缓存。

这样可以同时减少 VTW 重复计算、保证不同方法使用相同评估集,并避免训练/测试环境泄漏。

1. 主对比实验

对比对象:

- Single-PPO: 单星 PPO 基线;
- Indep-PPO: 多星独立 PPO,无协同 owner 分配;
- Vanilla MAPPO: 不使用高层 CVA 和 Top-K 的基础多星协同;
- CVA-MAPPO: 本文最终方法。

主要证明点:

- 相比 Single-PPO,多星方法能够提升可观测任务完成数;
- 相比 Indep-PPO,CVA-MAPPO 能显著降低重复观测率;
- 相比 Vanilla MAPPO,CVA-MAPPO 在大规模压力场景下有更好的调度效率、动态响应和负载均衡。

同时需要加入 owner 预分配效果对比:

- no-owner independent PPO:各卫星分开执行,最后统一统计,没有 owner 预分配,用于暴露重复观测和资源浪费;
- no-owner MAPPO:使用低层 MAPPO 协同,但不使用 owner 预分配,用于区分低层协同和高层 owner 的贡献;
- owner heuristic static:使用静态 owner 分配,验证预分配本身能否降低重复观测;
- owner CVA rolling:最终方案,验证价值感知 owner 和滚动修复是否进一步提升动态任务表现。

该对比用于证明:预先分配任务 owner 可以给多星系统提供稳定的任务责任边界,从源头降低重复观测率,避免多颗卫星围绕同一高价值任务重复消耗资源。

2. 候选动作空间消融

对比:

    full action
    Top-K = 64
    Top-K = 128
    Top-K = 256

证明目标:

- Top-K 不会明显损失任务完成率;
- Top-K 能降低训练时间和动作空间探索难度;
- Top-128 在完成率和效率之间取得较好平衡;
- 说明分层候选动作空间是大规模场景下必要的设计,不是简单工程加速。

3. CVA 分配模块消融

对比:

- heuristic_static;
- heuristic_rolling;
- cva_lstm_static;
- cva_lstm_rolling;
- cva_mlp/gru/transformer/set_transformer_rolling。

证明目标:

- rolling reallocation 是否提升动态任务完成和 owner 修复能力;
- CVA 是否优于纯启发式分配;
- 上下文编码器是否比无上下文或简单 MLP 更适合任务集合价值判断;
- CVA 与 rolling 是否具有互补性。

4. rescue 与 owner 稳定性指标

除完成率外,还应观察:

- duplicate_rate: 重复观测率;
- load_balance_cv: 负载变异系数;
- avg_dynamic_response_s: 动态任务平均响应延迟;
- n_replans: 重分配次数;
- owner_churn_rate: owner 切换率;
- stale_owner_rate: 失效 owner 比例;
- deadline_rescue_rate: deadline 救援率;
- n_feasible_tasks: 可观测任务数。

其中完成率应基于可观测任务数计算,不是全部任务数。这样可以避免把物理不可见任务计入失败,保证评价口径公平。

5. 大规模扩展性实验

建议进一步设计不同规模压力测试:

    6 星 / 12 星 / 18 星
    700 / 2100 / 3500 任务

证明目标:

- 当任务规模和卫星规模增大时,CVA-MAPPO 仍能保持较低重复率;
- Top-K 动作空间使 actor 维度不随全量任务数线性膨胀;
- 方法在大规模场景中比 full action MAPPO 更可扩展。


五、方法优势
------------

1. 动作空间可控

通过 Top-K candidate action space,低层 MAPPO 的动作维度由全量任务数决定变为由 K 决定。即使总任务数继续扩大,策略网络输出维度也可以保持稳定。

2. 协同机制清晰

高层 owner 分配从构造上减少重复观测,低层冲突解决和任务状态同步进一步保证多星不会重复消耗资源。

3. 兼顾稳定性和动态适应性

owner 分配提供稳定分工,rolling reallocation、deadline release 和 stale owner rescue 提供动态修复能力。

4. 研究逻辑完整

方法不是单纯堆叠技巧,而是围绕大规模动态多星调度的核心瓶颈形成分层架构:

    大规模任务集合 -> 高层价值分配 -> 候选动作压缩 -> 低层协同调度

5. 易于消融验证

每个模块都有清晰的消融接口:

- 是否使用 Top-K;
- Top-K 大小;
- 是否使用 CVA;
- 是否使用 rolling;
- 使用何种上下文编码器;
- 是否使用 deadline rescue;
- 是否使用 MAPPO 低层协同。


六、未来可扩展方向
------------------

1. 学习式高层分配器

当前 CVA 更接近上下文价值感知打分器。未来可以将高层分配器训练为真正的学习式策略,例如:

- 用监督学习拟合 oracle assignment;
- 用强化学习训练高层 owner 分配策略;
- 用 imitation learning 学习近似最优分配行为。

2. 图神经网络任务-卫星二分图建模

多星多任务天然构成 satellite-task bipartite graph。未来可以用 GNN 统一建模:

- 卫星节点;
- 任务节点;
- 可见窗口边;
- 边权包含质量、时间、deadline、机动成本。

这可以提升 CVA 对大规模任务集合结构的表达能力。

3. 自适应 Top-K

当前 K 是固定超参数。未来可以设计自适应 K:

- 高负载或高动态压力时增大 K;
- 当前可行动作少时减小 K;
- 根据任务密度、deadline 压力和卫星状态动态调节 K。

4. 更强的 rolling horizon 优化

当前 rolling reallocation 是轻量启发式/价值评分版本。未来可以结合:

- MPC;
- beam search;
- auction-based planning;
- learned reallocation trigger;
- uncertainty-aware replan。

5. 并行环境 rollout 与 GPU 批量训练

目前主要瓶颈仍在环境 rollout、VTW 计算和 action mask 构建。当前代码已经加入基础版 VTW 磁盘缓存和 MAPPO 训练 rollout 并行接口:

- `--vtw_cache_dir`:跨进程复用已计算过的卫星-目标可见时间窗口;
- `--train_env_workers`:将单个 MAPPO 训练迭代的 rollout 拆到多个环境进程中并行采样,再由主进程合并 buffer 统一更新。

未来还可以进一步:

- 批量化 action mask;
- 使用向量化环境提升 CPU 利用率。

6. 面向真实星座和真实任务流验证

未来可以接入更真实的轨道星座、任务到达过程和云量/天气约束,进一步验证方法在真实遥感调度场景中的适用性。


七、总结
--------

CVA-MAPPO 的核心贡献是将大规模多星动态任务调度拆解为高层价值分配和低层协同调度两个层次。高层 CVA 负责理解任务集合、分配 owner 并压缩候选动作空间;低层 MAPPO 在小规模候选集内进行时序调度;rolling reallocation 和 rescue 机制负责处理动态任务和 owner 失效。

该方案直接回应了大规模场景中 PPO/MAPPO 动作空间过大、多星重复观测、静态分配失效和动态响应不足等问题。通过主对比、Top-K 消融、CVA 消融和扩展性实验,可以系统证明该方法在完成率、重复率、动态响应、负载均衡和可扩展性方面的优势。
